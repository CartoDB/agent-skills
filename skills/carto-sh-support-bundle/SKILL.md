# CARTO Self-Hosted Support Bundle Analysis

Guides the analysis of support bundles generated by CARTO Self-Hosted (KOTS-based) installations. Bundles are extracted directories, not HAR files.

## When to Use

- A support engineer shares a support bundle directory path or extracted archive.
- A customer reports a Self-Hosted installation that is stuck, degraded, or showing unexpected status in the KOTS Admin Console.
- You need to triage a Self-Hosted issue before escalating to engineering.

## Steps

### Step 0: Check debug mode

Open `analysis.json`. If it contains `[ERROR] enable.debug.mode.for.carto.support`, debug mode is off and pod logs will be limited. **Stop and ask the customer to enable debug mode and resend the bundle before continuing.**

Docs: https://docs.carto.com/carto-self-hosted/maintenance/monitoring#enable-debug-mode

### Step 1: Check bundle timestamp

Note the bundle generation time vs the customer's last action (redeploy, config change, etc.). If the bundle predates the last action, ask for a fresh one — it won't reflect current state.

### Step 2: Overall app status

Read `kots/admin_console/app-info.json`:
- `app_status` — expected: `ready`. Common values: `missing`, `unavailable`, `degraded`.
- `sequence` — KOTS deploy sequence number.
- `deploy_source` — e.g. `"Config Change"`, `"Upstream Update"`.
- `k8s_version` / `k8s_distribution` — useful for context.

### Step 3: Requirement checks

Read `tenant-requirements-check/tenant-requirements-check.log` (JSON). Scan each check for `passed: false` or non-empty `errors`. Key validators:

| Validator | What it covers |
|---|---|
| `ServiceAccountValidator` | GCP SA permissions — most common root cause |
| `BucketValidator` | GCS/S3 bucket access |
| `RedisValidator` | Redis/Valkey connectivity |
| `PubSubValidator` | GCP Pub/Sub access |
| `EgressValidator` | Outbound internet access (packages, telemetry) |
| `CertificateValidator` | TLS/ingress cert validity |

> **Known noise**: LaunchDarkly `SDK key not found` failure is non-critical — feature flags are not required for core functionality.
>
> **Known noise**: Redis `"client is closed"` may be transient — verify Redis/Valkey pod health separately before treating as root cause.

### Step 4: Deployment state

Read `replicated/logs/<pod>/replicated.log`. Look for resource-level states (`ready` / `missing` / `unavailable`) with timestamps to understand *when* things went wrong and *which* deployments are affected.

Then read `cluster-resources/deployments/carto-self-hosted.json` and check desired vs ready vs available replicas per deployment.

### Step 5: Pod state

Read `cluster-resources/pods/carto-self-hosted.json`. Check `phase` and `containerStatuses` for `CrashLoopBackOff`, `ImagePullBackOff`, `OOMKilled`, or `Error`.

Check `registry/images.json` if present — `exists: false` on any image means pods may fail to schedule even if SA and config are correct.

### Step 6: Warning events

Read `cluster-resources/events/carto-self-hosted.json`. Filter by `type == "Warning"`. Common events: image pull failures, pod scheduling issues, OOMKill.

### Step 7: Pod logs

For any crashing container, read its logs from `namespace-carto-self-hosted-logs/<pod>/`. Grep for `error`, `warn`, `fail`, `exception`, `panic`.

For embedded-cluster (single-node) installs, pods live in the `kotsadm` namespace — see `references/bundle-structure.md` for path differences.

## Output Format

Produce a triage summary with these sections:

1. **Bundle info** — timestamp, KOTS sequence, distribution, debug mode on/off.
2. **App status** — `app_status` value and what it means.
3. **Failed checks** — list each failed requirement validator, the error message, and what it likely means.
4. **Deployment gaps** — which deployments are missing or not ready (desired vs actual).
5. **Pod errors** — container-level crash/pull errors.
6. **Root cause hypothesis** — most likely cause based on findings, with confidence.
7. **Recommended next steps** — specific actions for the customer or escalation path.

## Guardrails

- Do not diagnose from memory — read the actual files in the bundle before drawing conclusions.
- Do not tell the customer a fix "will" resolve the issue; use tentative language ("this may be caused by", "a likely fix is").
- Do not escalate to engineering for known patterns (SA permissions, Ingress Testing Mode, duplicate license) — resolve at support level first.
- Do not read pod logs before checking `analysis.json` for debug mode — logs are incomplete without it.
- Do not assume a standard KOTS layout for embedded-cluster bundles — the namespace and directory structure differ significantly (see `references/bundle-structure.md`).
